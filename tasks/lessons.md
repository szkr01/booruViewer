# Lessons

- 2026-03-08: collector の hot path に `sorted(set(...))` のような state compaction を入れると、画像処理と無関係に CPU スパイクして throughput が落ちる。
- 再発防止ルール:
  - state は探索ループ中に append/update のみ行い、unique/sort/truncate は保存時だけ実行する。
  - state 保存の責務は caller 側に散らさず、保存層で compaction を一元化する。

- 2026-03-08: 区間探索の改善で throughput 問題が出たら、まず hot path に DB 判定や区間探索が混ざっていないかを疑う。
- 再発防止ルール:
  - collector は「区間決定」と「区間消化」を分離し、消化中に DB 存在確認を入れない。
  - 長時間ジョブの 1 サイクルで複数区間をまたがらせず、1区間完了で次サイクルへ返す。

- 2026-03-08: legacy 資産の `vectors_raw.npy` は、拡張子が `.npy` でも実体が NumPy コンテナとは限らない。旧 ILEMB では raw float16 バイナリがその名前で置かれている場合がある。
- 再発防止ルール:
  - legacy ファイルはヘッダやサイズを確認し、拡張子だけで形式を決め打ちしない。
  - ベクトルファイルは `rows * dim * dtype_size` のサイズ整合も必ず確認する。

- 2026-03-08: ユーザーが「専用の運用入口」を求めているときは、内部サブコマンドが存在していても BAT/スクリプト入口まで実装し終える。
- 再発防止ルール:
  - 「バッチスクリプトはどうした」と言われたら、内部関数や既存入口で代用せず、入口ファイルの有無を先に確認する。
  - 初回移行と継続収集の責務が違う場合、ユーザー向け入口も別ファイルに分ける。

- 2026-03-08: 旧プロジェクト資産の互換移行では、ファイル名や shape だけでなく「Faiss 学習時/追加時の正規化」「ID 空間」「raw vector 保存形式」まで一致確認してから互換と言う。
- 再発防止ルール:
  - legacy index 再利用では、旧実装の build/search 両方を読んでベクトル前処理を照合する。
  - `.npy` と生バイナリ `.f16` は同一視せず、必要なら明示変換する。
  - Faiss の返す ID が `post_id` か `vec_idx` かを先に確定し、DB マッピング方式をそれに合わせる。

- 2026-03-08: Windows 環境で作成された `.venv` は WSL からそのまま利用できる前提で扱わない。
- 再発防止ルール:
  - WSL 側で検証するときは、Windows 側 `.venv` を前提にコマンドを組まない。
  - 依存実行確認が必要なら、WSL 用環境かシステム Python で可能な範囲に切り分ける。

- 2026-03-08: 要件に"運用フロー"が書かれている場合、初版で機能を1つのスクリプトに詰め込まず、移行(一度きり)と日次更新(反復実行)を分離して実装する。
- 再発防止ルール:
  - "migration"と"sync"は別エントリポイントにする。
  - `database.bat`の責務を段階化し、各段を独立実行可能にする。
- 2026-03-08: フロント互換を維持する場合、`/API/search`と`/API/tags`を空実装・限定実装のまま出さない。最低限のフォールバック検索を同時に実装する。
- 再発防止ルール:
  - UIが依存する主要APIはモックで埋めず、実データ経路（ローカル or 外部API）を必ず用意する。
  - テキスト検索とprefix補完はセットで検証する。
- 2026-03-08: 「booru検索は使わない」という要件がある場合、外部Danbooru検索APIへのフォールバックを実装しない。
- 再発防止ルール:
  - データ供給経路を「ローカル完結」か「外部依存あり」かで明示し、設計時点で固定する。
  - 要件と逆方向のフォールバックは追加しない。
- 2026-03-08: `docs/`は参照資料であり、実行時依存に入れない。
- 再発防止ルール:
  - 初期`config.json`にサンプル資料パスを直書きしない。
  - 実行パスは`data/`配下またはユーザー設定値のみを使う。
- 2026-03-08: 「database.batで自動構築」と要件化されている場合、入力データ前提（手動配置parquet前提）で終了させない。必ずデータ取得・欠損補完・状態管理まで含める。
- 再発防止ルール:
  - バッチ入口は「No input found」で正常終了せず、次の取得/補完ステップへ遷移させる。
  - 方針文に「最新」「未構築」がある場合、`latest sync` と `backfill` を別ステップで設計する。
- 2026-03-08: Windows前提プロジェクトで実行確認するときは、WSL側の`uv run`検証を優先しない。まず `.bat` エントリポイント（PowerShell/CMD）での動作確認手順を提示する。
- 再発防止ルール:
  - 実行検証コマンドはユーザー運用環境（今回: Windows）に合わせる。
  - WSL/Unix系の検証は「補助確認」として明示し、先に本番運用経路を通す。
- 2026-03-08: ユーザーから運用方針の再確認指示が入ったら、改善提案を先に出さず、まず `AGENTS.md` の再読と準拠を完了してから次アクションを示す。
- 再発防止ルール:
  - 方針逸脱が発生した直後は、ドキュメント変更提案より先に運用ルール準拠の回復を優先する。
- 2026-03-08: GPU前提プロジェクトでは、セットアップ後にCUDA有効性を必ず自動検証し、実行スクリプトで依存再同期（CPU版上書き）を起こさない。
- 再発防止ルール:
  - `setup.bat` は CUDA `torch.cuda.is_available()` 検証を必須化する。
  - `.bat` 実行時は `uv run --no-sync` を使い、ロック同期で環境が書き換わる経路を遮断する。
- 2026-03-08: 大規模バッチ（100万件級）では、処理速度より先に小粒度IO（state保存・小分割parquet・毎バッチfsync）がボトルネックになる。
- 再発防止ルール:
  - 投稿ごとの固定 `sleep` を入れず、`429/5xx` ベースの適応バックオフにする。
  - 状態保存とチェックポイントは件数/バッチ間隔で間引き、毎ループ書き込みを避ける。
  - 耐障害性機能（`fsync`）はデフォルト常時ONにせず、運用規模に応じて切替可能にする。
- 2026-03-08: 終了しない前提の長時間ジョブでは、IO最適化時に停止耐性を落とさない。件数だけでなく時間ベースのチェックポイントを併用する。
- 再発防止ルール:
  - state保存条件は「N件ごと」だけにせず「T秒ごと」も必ず入れる。
  - stateファイルは原子的保存（tmp + replace）にして、中断時の破損を防ぐ。
- 2026-03-08: 収集系の要求に「起動中ずっと自動で続ける」「小ファイルを増やさない」「通信/GPUの両方に合わせて速度を出す」が含まれる場合、単発バッチ実行として実装しない。
- 再発防止ルール:
  - `database.bat` のような収集入口は、要件に常駐性があるなら最初からループ/デーモンとして設計する。
  - 出力フォーマットは append/rolling を前提にし、1バッチ1ファイルの実装を避ける。
  - ダウンロードと埋め込みは分離し、少なくとも通信側は並列化と自動調整を入れる。
- 2026-03-08: ユーザーが明示的に必要としている機能（今回: HTTP/2）について、依存不足を理由に機能自体を外す方向へ振らない。
- 再発防止ルール:
  - まず依存を正しく宣言・導入する。
  - フォールバックを入れる場合も「機能を諦める前提」ではなく「暫定回避」であることを明示する。
- 2026-03-08: 常駐収集の要件に反する人工的な処理上限（`max_posts` など）を追加しない。
- 再発防止ルール:
  - 収集器の停止条件は要件にあるものだけにする。
  - サイクル制御が必要なら「一周の上限件数」ではなく、明示的な運用仕様としてユーザー確認を取る。
- 2026-03-08: WSL から Windows 前提の `.venv` を直接変更しない。ローカル実行環境の更新はユーザー運用経路に委ねる。
- 再発防止ルール:
  - 依存修正は `requirements.txt` / `pyproject.toml` / `uv.lock` までに留める。
  - 実環境の `uv sync` / `setup.bat` 実行は、ユーザーの Windows 側運用フローとして案内する。
- 2026-03-08: ユーザーが実装方針の原文確認を求めたら、先に `docs/impl_request.md` を読み直してから検証・提案を続ける。
- 再発防止ルール:
  - 方針確認前にログや実装だけで評価を固めない。
  - 「何を検証すべきか」は `impl_request.md` の文言に合わせて再定義する。
- 2026-03-08: ユーザーが保存条件を「時間だけ」と指定したら、件数基準の保存トリガーを残さない。
- 再発防止ルール:
  - `*_save_every` のような件数設定が残っていないか実装と設定の両方を確認する。
  - バッチ flush と state 保存は別物として扱い、保存条件を勝手に増やさない。
- 2026-03-08: ユーザーが「1つのアルゴリズムで十分」と明示したときは、段階ごとの別スクリプトを増やさず単一 collector に統合する。
- 再発防止ルール:
  - 収集ロジックを「latest」「backfill」などの別フェーズに分ける前に、単一の探索器で完結できないかを先に検討する。
  - シンプルさと保守性が要件にある場合、同じ責務の複数エントリポイントを作らない。
- 2026-03-08: `Ctrl+C` で収集を止める要件がある場合、バッチ入口は `130` を単純失敗扱いせず、flush 済みデータの index 反映まで完了させる。
- 再発防止ルール:
  - `.bat` / シェル入口で中断コードの扱いを確認する。
  - 収集と index 反映が分離されているなら、中断時も後段を流す経路を用意する。
- 2026-03-08: 検索用ベクトルと分類用特徴は同一視しない。`tags_from_id` のような逆変換では検索用正規化ベクトルを流用しない。
- 再発防止ルール:
  - Faiss投入前の正規化は index 用コピーに限定し、生特徴の保存を壊さない。
  - 分類ヘッドへ戻す経路は、生特徴を優先し、重い再推論を勝手に常用経路へ入れない。
- 2026-03-08: ユーザーが「重い処理を入れるな」と明示していなくても、API の常用経路を再推論ベースへ変える前にコストを精査する。
- 再発防止ルール:
  - 既存データ不整合の修正であっても、まず軽量な保存データ経路を優先する。
  - 再推論は明示要求か最終フォールバックに限定する。
- 2026-03-08: サジェスチョン UI の hover 表示更新で、クリック対象要素を再生成しない。
- 再発防止ルール:
  - ポインタ操作中の候補リストは `mouseenter` や `mousemove` で全再描画しない。
  - 選択状態の更新は DOM 差し替えではなく class 切り替えで済ませ、クリックイベントが落ちないことを確認する。
- 2026-03-08: ユーザーが「docs/repo を先に読め」「今回はフロントだけ」と明示したら、通信責務の確認前にバックエンド案へ広げない。
- 再発防止ルール:
  - 参照 repo がある修正では、現行コードより先に `docs/repo` の同責務実装を確認する。
  - 修正範囲が「フロントだけ」と指定されたら、バックエンド変更は提案対象にも含めない。
- 2026-03-08: 通信量制御の指摘が入ったら、「何もしない時に何が飛ぶべきか」を先に固定する。
- 再発防止ルール:
  - 一覧表示と詳細表示の通信を混ぜない。詳細用 HTML 取得は詳細 open 時だけに限定する。
  - lazy load 要件がある場合、カード生成時に即 `src` を入れず、observer 経由で読み込む。
- 2026-03-08: すでに動いている表示経路がある状態で通信最適化する場合、その経路を消してから置き換えない。
- 再発防止ルール:
  - 一覧サムネのような既存成功経路は維持したまま、追加要件の範囲だけを変える。
  - 「余計なことをするな」という修正では、最小差分で戻す対象を先に特定する。
- 2026-03-08: URL 規則が保存データに明示されていないとき、解像度違いを同じパターンで復元できると仮定しない。
- 再発防止ルール:
  - 既知なのが `720x720` のみなら、その範囲だけを使う。
  - 別解像度の URL は実データか一次ソースで確認できるまで生成しない。
- 2026-03-08: ユーザーの「D&D が壊れた」は、アップロード受け側ではなく一覧カードのドラッグ開始側を指している場合がある。対象経路を特定せずに直さない。
- 再発防止ルール:
  - D&D 不具合では「外からドロップ」「カードを外へドラッグ」のどちらかを先に確定する。
  - `docs/repo` に同機能がある場合、イベントハンドラで渡している payload まで一致確認する。
